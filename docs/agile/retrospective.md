# Retrospective: 4주 경량 애자일 회고 템플릿

## 회고 목적

이 회고는 1인 운영자가 Codex와 함께 빠르게 구현한 결과를 점검하기 위한 것이다. 사람 기준의 작업량 추정보다 Codex를 통한 구현, 검증, 수정 반복 속도를 기준으로 삼는다.

이번 계획에서 다루는 범위는 다음 두 가지다.

- 제안 1: 연구 주제 허브
- 제안 2: 자료 해설형 아카이브

제안 3: 공개 질의 기반 안내 시스템은 현재 계획에 포함하지 않는다.

## 스프린트별 회고 질문

각 스프린트 종료 후 아래 질문에 짧게 답한다.

### 1. 유지할 것

- 어떤 작업 방식이 Codex와 잘 맞았는가?
- 어떤 프롬프트나 검증 절차가 재사용 가능했는가?
- 어떤 컴포넌트나 데이터 구조가 안정적으로 작동했는가?

### 2. 개선할 것

- 어떤 요구사항이 모호해서 재작업이 발생했는가?
- 어떤 UI가 모바일에서 예상보다 많이 흔들렸는가?
- 어떤 데이터 구조가 임시 mock에 너무 의존했는가?

### 3. 중단할 것

- 불필요하게 큰 범위로 번진 작업은 무엇인가?
- 지금 하지 않아도 되는 기능은 무엇인가?
- 운영 부담을 늘리는 설계는 무엇인가?

### 4. 다음 스프린트 조정

- 다음 주에 반드시 끝내야 할 1순위는 무엇인가?
- 다음 주에 미뤄도 되는 것은 무엇인가?
- 사용자 원문이 필요한 콘텐츠는 무엇인가?

## 전체 4주 회고 항목

4주 종료 후 아래 항목을 기준으로 판단한다.

## 제품 관점

- 홈페이지가 게시판 중심에서 주제/자료 탐색 중심으로 실제로 이동했는가?
- 방문자가 처음 들어왔을 때 다음 행동을 명확히 알 수 있는가?
- 연구소의 신뢰감, 학술성, 전문성이 강화되었는가?

## 운영 관점

- 1인 운영자가 관리할 데이터가 과도하게 늘지 않았는가?
- mock 데이터와 DB 데이터의 경계가 명확한가?
- 관리자 기능 없이도 당분간 수동 운영이 가능한가?

## 기술 관점

- `npm run lint`와 `npm run build`가 안정적으로 통과하는가?
- Server Component와 Client Component 경계가 불필요하게 넓어지지 않았는가?
- Prisma 모델 변경 없이 가능한 범위와 모델 변경이 필요한 범위가 분리되었는가?

## 디자인 관점

- 기존 보라색 브랜드 디자인 시스템과 일관되는가?
- 카드, 링크, CTA가 반복 가능한 패턴으로 정리되었는가?
- 데스크톱과 모바일에서 가로 overflow가 없는가?

## 최종 판단 기준

다음 조건을 만족하면 4주 계획은 성공으로 본다.

- 연구 주제 허브가 홈과 상세 경로에서 작동한다.
- 자료 해설형 아카이브가 목록과 상세 페이지에서 작동한다.
- 제안 3 FAQ 기능 없이도 탐색 경험이 충분히 개선된다.
- 주요 공개 페이지가 모바일과 데스크톱에서 깨지지 않는다.
- 운영자가 혼자서 콘텐츠를 추가하고 수정하는 흐름을 이해할 수 있다.
