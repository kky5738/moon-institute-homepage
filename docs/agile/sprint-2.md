# Sprint 2: 연구 주제 상세 페이지와 카테고리 연결

## 스프린트 목표

연구 주제 허브를 단순 홈 섹션이 아니라 실제 탐색 구조로 확장한다. 각 주제별 상세 페이지를 만들고, 기존 게시글 카테고리와 연결해 공지, 자료, 영상, 문의 CTA가 한 화면에서 연결되는 경험을 구현한다.

## 기간

- 1주
- Codex 기준 작업 속도: 라우팅, 서버 컴포넌트 데이터 조회, 반응형 UI, 검증까지 포함한다.

## 작업 범위

- 제안 1: 연구 주제 허브
- 제안 2와 연결될 수 있도록 자료 목록 영역을 포함한다.
- 제안 3 FAQ/공개 질의 기반 안내 시스템은 제외한다.

## 주요 산출물

- `/topics` 또는 `/topics/[slug]` 라우트 설계 및 구현
- 주제별 상세 페이지
- 주제별 관련 게시글/자료 목록
- 주제 페이지에서 문의/참여 신청으로 이동하는 CTA
- 홈 주제 카드에서 상세 페이지로 연결

## Codex 작업 단위

1. 라우트 이름 결정: `/topics`와 `/topics/[slug]`를 우선 검토
2. mock 주제 데이터와 기존 `Category`/게시글 데이터 연결 방식 정의
3. 주제 목록 페이지 또는 주제 상세 페이지 구현
4. `getPublishedPosts` 또는 별도 helper로 주제 관련 게시글 필터링
5. 상세 페이지에 관련 공지, 관련 자료, 영상 안내, 문의 CTA 배치
6. 빈 데이터 상태 UI 추가
7. `npm run lint`, `npm run build`, Playwright 검증

## 완료 기준

- 홈 주제 카드에서 주제 상세 페이지로 이동할 수 있다.
- 주제 상세 페이지는 제목, 설명, 관련 자료/공지, CTA를 포함한다.
- DB에 게시글이 없거나 카테고리 연결이 약해도 빈 상태가 안정적으로 표시된다.
- 기존 `/notices`, `/materials`, `/contact` 흐름은 유지된다.
- 데스크톱과 모바일에서 레이아웃이 깨지지 않는다.

## 제외 범위

- 관리자 주제 편집 기능
- 신규 `Topic` Prisma 모델
- 자료 추천 관계 테이블
- FAQ 또는 공개 문의 콘텐츠 기능

## 리스크와 대응

- 리스크: 카테고리와 주제의 의미가 섞일 수 있다.
- 대응: 초기에는 “주제 허브 mock 데이터”와 “게시글 카테고리”를 느슨하게 연결하고, 실제 운영 경험 후 모델 분리를 판단한다.

- 리스크: 상세 페이지가 빈 콘텐츠로 보일 수 있다.
- 대응: 빈 상태에서도 추천 경로와 문의 CTA를 제공한다.
