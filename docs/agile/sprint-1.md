# Sprint 1: 연구 주제 허브 기반 설계와 홈 진입점

## 스프린트 목표

연구소 홈페이지의 정보 구조를 게시판 중심에서 주제 중심으로 전환하기 위한 첫 기반을 만든다. 실제 DB 모델 변경은 최소화하고, 기존 `Category`, `Post`, `site-content.ts` mock 데이터를 활용해 방문자가 “연구 주제”를 먼저 탐색하는 흐름을 홈에 도입한다.

## 기간

- 1주
- Codex 기준 작업 속도: 설계, 구현, 문서화, Playwright 검증까지 한 스프린트 안에서 완료하는 것을 기준으로 잡는다.

## 작업 범위

- 제안 1: 연구 주제 허브
- 제안 2는 이번 스프린트에서 직접 구현하지 않고, 주제 허브가 자료 아카이브와 연결될 수 있도록 정보 구조만 고려한다.
- 제안 3 FAQ/공개 질의 기반 안내 시스템은 제외한다.

## 주요 산출물

- 홈에 “연구 주제 허브” 섹션 추가
- `site-content.ts`에 주제 mock 데이터 추가
- 각 주제 카드에서 관련 자료, 공지, 영상, 문의 CTA로 이동할 수 있는 링크 구조 설계
- 기존 홈 구조와 충돌하지 않는 반응형 UI

## Codex 작업 단위

1. 현재 홈 컴포넌트 구조 확인
2. 주제 허브 mock 데이터 타입과 배열 작성
3. `HomeLandingSections`에 주제 허브 섹션 추가
4. 카드 UI를 기존 디자인 시스템의 `primary`, `primary-dark`, `accent`, `surface`, `border`, `muted` 토큰으로 구성
5. 모바일/데스크톱 레이아웃 확인
6. `npm run lint`, `npm run build`, Playwright 스크린샷 검증

## 완료 기준

- 홈에서 연구 주제 허브가 명확히 보인다.
- 주제 카드가 최소 3개 이상 표시된다.
- 각 카드에는 제목, 짧은 설명, 연결 CTA가 있다.
- 데스크톱과 모바일에서 가로 overflow가 없다.
- 기존 Hero, 영상 콘텐츠, 활동/자료/공지, CTA 섹션이 깨지지 않는다.
- `npm run lint`와 `npm run build`가 통과한다.

- 현재 주제명은 `주제1`, `주제2`, `주제3` 임시 문구다.

- 실제 연구 주제명과 설명 원문이 확정되면 Sprint 2 또는 이후 콘텐츠 정리 단계에서 교체한다.

## 리스크와 대응

- 리스크: 주제 설명이 임의 사실처럼 보일 수 있다.
- 대응: 주제 문구는 운영 방향과 탐색 구조 중심으로 작성하고, 인물·역사·사상 설명은 확정 원문 전까지 피한다.

- 리스크: 홈이 너무 길어질 수 있다.
- 대응: 섹션 간 우선순위를 조정하고 카드 수를 제한한다.
